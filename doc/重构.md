# 整体重构

## 决策摘要

- 全面转为 TypeScript 技术栈（一次性替换，不走长期双栈）。
- 暂时不实现 `openclaw` 与 `screen-agent`，先聚焦本地 Recall 主链路。
- 后端框架固定为 **Fastify**，对外接口固定为 **HTTP REST**。
- OCR 路线固定为 **onnxruntime-node 自建**。
- 前端固定为 **React + Vite + Tailwind**。
- 数据库固定为 **SQLite**。
- 多 agent 协作入口固定为根目录唯一 `AGENT.md`，所有 agent 启动第一步必须读取该文件。
- skills 目录迁移到 `.agents/skills`，不依赖默认 skills 发现。

---

## Agent Workflow（L1/L2）

### 一次性替换原则

- 根目录 `AGENT.md` 为唯一 workflow 入口，所有 agent 启动第一步必须读取。
- skills 只允许从 `.agents/skills/` 加载，禁止默认路径自动发现回退。
- 旧流程全量废弃，不保留历史入口与历史协作目录。

### 新入口

- `/l1`：需求澄清、方案设计、任务拆解、用户确认（支持小改动直接执行）。
- `/l2`：任务认领、开发实现、测试、代码审查、合并收口。

### tasks 目录与状态

- 任务看板固定为：
  - `tasks/pending`
  - `tasks/active`
  - `tasks/blocked`
  - `tasks/done`
  - `tasks/cancelled`
- 任务文件命名：`NNN.md`（三位递增编号）。
- 状态以目录为准，`status` 字段仅用于冗余一致性检查。

### 任务文件规范（精简字段）

front matter 固定字段：

- `id`
- `title`
- `status`
- `branch`
- `worktree`
- `created_at`
- `updated_at`

正文必须详细包含：

1. `背景`
2. `目标`
3. `范围`
4. `实施计划（分步骤）`
5. `验收标准`
6. `测试要求`

### L1 流程（需求入口）

1. 读取 `AGENT.md` 与相关架构文档。
2. 与用户确认范围、数据流、验收标准。
3. 小改动直接执行；中大改动先在 `tasks/pending/` 产出任务。
4. 拆任务按“可独立交付”粒度，任务正文必须写清实施计划。

### L2 流程（执行入口）

1. 从 `tasks/pending/` 认领任务并迁移到 `tasks/active/`。
2. 创建独立分支 + worktree 开发（每个 active 任务唯一绑定）。
3. 按任务中的测试要求执行验证并完成代码审查。
4. 完成后合并回主线并迁移到 `tasks/done/`。
5. 阻塞时迁移到 `tasks/blocked/`，记录 `blocked_reason` 与解除条件。

### Worktree 规则

- 多 agent 并行开发必须使用 `git worktree`。
- 单任务单分支单 worktree，不允许复用活跃工作目录。
- 主分支只做任务创建、状态同步、合并收口，不承担并行实现开发。

## 技术路线与模块划分（TS 版）

### 总体技术选型

- 语言：TypeScript（前后端统一）。
- 后端：Fastify。
- 前端：React + Vite + Tailwind。
- 数据库：SQLite（本地优先）。
- OCR：onnxruntime-node。

### 模块划分

1. `trigger`
   - 屏幕变化检测
   - 强制间隔计时
   - 对外发出“需要截图”事件

2. `capture`
   - 接收触发事件执行截图
   - 图片落盘（文件系统）
   - 写入截图元数据到数据库

3. `resource-monitor`
   - 监控 GPU/CPU 使用率
   - 输出“空闲/繁忙”状态给 OCR 调度器

4. `ocr-worker`
   - 批量读取待处理截图
   - 调用 onnxruntime-node 进行 OCR
   - 回写 OCR 文本和状态

5. `api`
   - 暴露 REST 接口给前端与其他 agent
   - 提供 OCR 查询、截图查询、summary 写入与查询

6. `frontend`
   - 截图浏览
   - 配置编辑（动态配置写入数据库）
   - summary 查询视图

7. `main`
   - 组装 trigger / capture / monitor / ocr-worker / api
   - 负责模块生命周期与启动顺序

### 核心数据流

1. `trigger` 判定达到阈值或强制间隔 -> 发截图事件。
2. `capture` 截图并保存文件 -> 写入 `screenshots` 记录（`ocr_status=pending`）。
3. `resource-monitor` 提供负载状态。
4. `ocr-worker` 在资源允许时批量 OCR -> 更新 `ocr_text/ocr_status`。
5. `api` 对外提供时间范围查询、单图查询、summary 写入与查询。

## 数据存储设计

### 配置分层

- `config.json`：预留为**静态配置**（启动时加载，低频改动）。
- 数据库 `settings`：保存**动态运行配置**（支持热更新、实时生效）。

动态配置（数据库）至少包含：

- 截图变化阈值（CHANGE_THRESHOLD）
- OCR 阈值（OCR_THRESHOLD）
- OCR 批大小（OCR_BATCH_SIZE）
- GPU 繁忙阈值（GPU_USAGE_THRESHOLD）
- CPU 繁忙阈值（CPU_USAGE_THRESHOLD）

### 图片存储策略

- 图片存文件系统，不存 SQLite BLOB。
- 数据库主键使用自增 `id`。
- `screenshots` 表只存元数据与文件路径。

字段：

- `id`（INTEGER PRIMARY KEY AUTOINCREMENT）
- `captured_at`
- `file_path`
- `ocr_text`
- `ocr_status`（pending/done/error）
- `window_title`
- `process_name`
- `phash`
- `created_at`

目录：

- `screenshots/YYYY-MM-DD/HH/{id}.jpg`

### Summary 存储

- `summaries` 表：`id/start_time/end_time/summary/activity_type/created_at`
- 提供按时间段查询与写入。

## API 基线

- `GET /api/ocr?start_time=&end_time=&limit=`
- `GET /api/screenshot/:id/image`
- `POST /api/summaries`
- `GET /api/summaries?start_time=&end_time=` 或 `?hours=`
- `GET /api/config` / `POST /api/config`（动态配置读写数据库）

## 验收标准

- 所有核心链路（触发 -> 截图 -> OCR -> 查询）运行在 TS/Fastify 实现。
- 根 `AGENT.md` 成为多 agent 唯一入口，`.agents/skills` 生效。
- 动态运行配置全部入库，静态配置与动态配置边界清晰。
- 图片文件落盘 + SQLite 元数据模型跑通，截图主键为自增 id。
- REST API 能满足前端浏览与外部 agent 调用。
