## 背景

当前屏幕变化检测使用截图字节的 SHA 截断哈希，导致微小字节波动也会被判定为变化；同时 `CHANGE_THRESHOLD`、`FORCE_INTERVAL` 等配置写入 settings 后未被监控器动态读取，出现截图频率异常且配置不生效的问题。

## 目标

1. 将屏幕检测哈希切换为真正的感知哈希，降低无意义高频触发。
2. 让 `ScreenMonitor` / `TimeMonitor` 支持动态配置，并在配置变更后快速生效。
3. 明确并落地“后端事件即时刷新 + 轮询兜底”的动态生效机制。

## 范围

- 包含：
  - 检测哈希算法替换（感知哈希 + 海明距离阈值判定）。
  - `ScreenMonitor` 动态读取 `CHANGE_THRESHOLD`、`SCREEN_CHECK_INTERVAL`（新增）。
  - `TimeMonitor` 动态读取 `FORCE_INTERVAL`。
  - 配置更新后后端事件通知链路（`POST /api/config` -> Engine -> monitor reload）。
  - 对应单元测试/集成测试补充与回归验证。
- 不包含：
  - 前端页面结构重做。
  - 新增远程配置中心或跨进程消息系统。

## 实施计划（分步骤）

1. 梳理当前截图检测路径，替换 `current_screen_hash` 的实现为感知哈希（建议 dHash），保持现有阈值比较接口不变。
2. 在 settings 增加 `SCREEN_CHECK_INTERVAL` 默认值，并为两个 monitor 增加配置读取与类型解析、异常兜底逻辑。
3. 扩展事件模型与 Engine：新增配置更新事件；在配置 API 保存成功后发布事件；Engine 收到后调用 monitor 的 `reload_config()` 立即生效。
4. 保留监控循环中的轻量配置刷新作为兜底，确保非前端入口修改 settings 也能在下个周期生效。
5. 增补测试：哈希行为测试、动态配置即时/延迟生效测试、端到端频率回归验证。

## 验收标准

- 稳态场景下截图频率明显下降，不再接近固定 3 秒一张。
- 前端配置页修改阈值/间隔后，无需重启服务即可生效。
- 非前端路径（脚本或直接改 settings）修改配置后，最多一个轮询周期内生效。
- 关键测试通过，且无现有 API 兼容性回归。

## 测试要求

- 单元测试：
  - 感知哈希与海明距离判定边界（无变化/轻微变化/明显变化）。
  - monitor 动态读取与非法配置回退默认值。
- 集成测试：
  - `POST /api/config` 后 monitor 立即应用新配置。
  - 运行一段采样窗口，验证截图间隔分布符合预期。
