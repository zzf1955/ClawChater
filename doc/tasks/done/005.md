## 背景

截图入库后仍需 OCR 才能形成可检索文本；重构方案要求由 `resource_monitor` 在系统空闲时驱动 `ocr_worker` 批处理。

## 目标

实现资源监控与 OCR worker，使 `pending` 截图可按批次转为 `done/error` 并回写文本。

## 范围

- 包含：CPU/GPU 阈值判断、`resource_available` 事件、pending 批量拉取、OCR 推理适配、状态回写。
- 不包含：OCR 模型训练、摘要生成逻辑、前端 OCR 可视化优化。

## 实施计划（分步骤）

1. 实现 `resource_monitor.py`，周期采样资源并按配置阈值发出空闲事件。
2. 实现 `ocr_worker.py`，按 `OCR_BATCH_SIZE` 拉取 pending 记录并执行批处理。
3. 封装 OCR 调用与错误处理，确保单条失败不阻断整批。
4. 回写 `ocr_text` 和 `ocr_status`，输出可追踪日志与统计信息。

## 验收标准

- 资源空闲时可触发 OCR worker 执行。
- pending 记录可按批次被处理并更新状态。
- 成功记录写入文本并标记 `done`，失败记录标记 `error`。
- 无待处理记录时 worker 快速返回，不造成空转负担。

## 测试要求

- 单元测试覆盖批次切分、阈值判断、状态回写。
- 使用 mock OCR 引擎做集成测试，验证成功/失败混合批次行为。
- 增加稳定性测试，验证长时间轮询下无明显内存泄漏。
