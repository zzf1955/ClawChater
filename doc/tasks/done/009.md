## 背景

在验证 `recall` 新架构时，发现“屏幕检测 -> 截图 -> OCR”触发链路排查困难，缺少可定位问题的运行日志。

## 目标

提升运行期可观测性：在时间触发、事件发布、截图执行、OCR 批处理等关键链路输出 debug/info，并将日志落盘，便于排查触发异常。

## 范围

- 包含：
  - App 启动时统一日志配置（控制台 + 文件）。
  - Screen/Time/Resource monitor 的触发日志。
  - EventBus 发布日志、Engine 事件处理日志。
  - Capture/OCR 执行结果日志。
  - notice 文档补充调试日志入口说明。
- 不包含：
  - 数据库迁移脚本与迁移演练。
  - 新增业务功能。

## 实施计划（分步骤）

1. 梳理 `app -> engine -> monitors -> capture/ocr` 现有触发路径，补齐日志断点。
2. 增加统一文件日志能力，支持环境变量配置日志级别和日志文件路径。
3. 运行核心回归测试，确认日志增强不影响原有行为。
4. 根据用户要求移除数据库迁移相关脚本与文档引用，完成收尾。

## 验收标准

- 日志文件可生成并持续写入。
- 关键触发点（screen/time/resource/capture/ocr）均可在日志中观测到。
- 回归测试通过，未引入功能回归。
- 数据库迁移相关改动已按用户要求移除。

## 测试要求

- 运行核心测试集：`test_engine/test_monitors/test_capture_engine/test_ocr_worker/test_api`。
- 启动服务后通过 `tail -f recall/data/logs/recall.log` 观察触发链路日志。
