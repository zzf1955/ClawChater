## 目标

支持多台机器协作截图：slave 机器只截图并通过 syncthing 同步到 host，host 端自动导入并 OCR。

## 范围

| # | 内容 | 文件 |
|---|------|------|
| 1 | Slave 轻量截图客户端 | `recall/slave.py`（新建） |
| 2 | Host 端 IncomingWatcher | `recall/services/incoming_watcher.py`（新建） |
| 3 | 配置项扩展 | `recall/config.py`（修改） |
| 4 | Engine 集成 IncomingWatcher | `recall/services/core/engine.py`（修改） |

## 实施计划（分步骤）

### 步骤 1: Slave 客户端 (`recall/slave.py`)

- 轻量独立脚本，`python -m recall.slave` 运行
- 复用 `CaptureService._default_screenshot_provider` 截图逻辑
- 定时截图，写入 syncthing 同步目录
- 每张截图产出两个文件：
  - `{device_id}_{timestamp}.jpg`
  - `{device_id}_{timestamp}.json`（sidecar 元数据）
- JSON 包含 device_id, captured_at, window_title, process_name, platform
- 支持 macOS + Windows
- 屏幕变化检测（phash），只在变化时截图
- 配置：RECALL_DEVICE_ID, RECALL_SYNC_DIR, RECALL_CAPTURE_INTERVAL

### 步骤 2: Host IncomingWatcher (`recall/services/incoming_watcher.py`)

- 轮询监控 incoming 目录中的新 .jpg 文件
- 读取配套 .json sidecar
- 移动截图到标准 `data/screenshots/YYYY-MM-DD/HH/` 路径
- 调用 `insert_screenshot` 导入数据库，ocr_status=pending
- 实现 `run()/stop()` Monitor 协议

### 步骤 3: Engine 集成

- config.py 添加 INCOMING_DIR 配置
- Engine 启动时若配置了 INCOMING_DIR，启动 IncomingWatcher
- IncomingWatcher 作为额外 monitor 加入 _monitors 列表

## 验收标准

- slave 端运行后，截图 + json 文件正确写入 sync_dir
- host 端 IncomingWatcher 检测到新文件后自动导入 DB
- 已有 OCR 流程自动处理导入的截图
- 现有功能无回归

## 测试要求

- `python -m pytest recall/tests/ -v` 全部通过
- 手动测试 slave 截图输出
- 手动测试 IncomingWatcher 导入流程
