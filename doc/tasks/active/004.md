## 背景

主链路的第一段是“检测触发 -> 执行截图 -> 写入 pending 记录”，这是后续 OCR 与查询能力的前置条件。

## 目标

实现 `screen_monitor`、`time_monitor`、`capture` 三个模块，并通过 Engine 在触发事件时完成截图入库。

## 范围

- 包含：屏幕变化检测（phash）、强制间隔触发、截图落盘、元数据入库、计时器重置。
- 不包含：资源监控、OCR 推理、前端展示。

## 实施计划（分步骤）

1. 实现 `screen_monitor.py`，按固定频率采样并基于阈值发出 `screen_change`。
2. 实现 `time_monitor.py`，按强制间隔发出 `force_capture` 并支持重置。
3. 实现 `capture.py`，完成截图文件写入与 `screenshots` 表插入（`ocr_status=pending`）。
4. 在 Engine 中接入上述模块，打通触发到入库链路。

## 验收标准

- 屏幕变化或超时均可触发截图任务。
- 截图文件按约定路径写入 `data/screenshots/YYYY-MM-DD/HH/`。
- 数据库新增记录字段完整，初始状态为 `pending`。
- 每次成功截图后，强制计时器会被正确重置。

## 测试要求

- 单元测试覆盖 phash 阈值判断与时间触发逻辑。
- 集成测试覆盖一次完整“触发 -> 截图 -> 入库”流程。
- 增加异常测试：截图失败、写盘失败、入库失败时的回退与日志。
