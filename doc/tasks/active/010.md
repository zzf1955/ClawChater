## 背景

当前项目前后端分离运行：FastAPI 提供 `/api`，前端由 Vite 单独提供页面。用户希望在非 dev 模式下，可由单一服务直接在浏览器访问完整页面与 API，减少启动复杂度。

## 目标

实现“FastAPI 同时提供 API + 前端静态页面”的一体化运行模式：构建前端后，仅启动 FastAPI 即可完成页面访问与业务接口调用。

## 范围

- 包含：
  - FastAPI 挂载前端 `dist` 静态资源（`index.html`、`assets/*`）。
  - 增加 SPA fallback（非 `/api` 路由返回 `index.html`）。
  - 梳理并固定生产模式下前端 API 访问策略（同源或可配置）。
  - 补充一体化启动文档与最小验证步骤。
- 不包含：
  - 前端 UI 功能重构。
  - 反向代理（Nginx/Caddy）生产部署编排。
  - 鉴权、HTTPS、容器化改造。

## 实施计划（分步骤）

1. 梳理现有入口：确认 `recall/app.py`、`recall/api/routes.py`、`recall/frontend/dist` 的职责边界与路由冲突点。
2. 在 FastAPI 中新增静态托管与 fallback 逻辑：确保 `/api/**` 保持原行为，页面路由（如 `/screenshots`）可直接打开。
3. 校准前端 API 基地址策略：保证 build 后由 FastAPI 托管时，浏览器请求能正确命中 `/api/**`。
4. 增加运行说明：提供“build 前端 + 启动 FastAPI + 浏览器访问”的命令与注意事项。
5. 增加最小回归验证：覆盖首页加载、静态资源返回、关键 API 可用、前端页面内请求成功。

## 验收标准

- 执行前端 build 后，仅启动 FastAPI，可在浏览器正常打开前端页面。
- 直接访问前端子路由（如 `/screenshots`）不返回 404，而是正确渲染 SPA 页面。
- 页面内截图查询、配置查询等请求能命中 `/api/**` 并成功返回。
- `/api/**` 接口行为与现有测试基线一致，不被静态路由覆盖。
- README/文档中有清晰的一体化启动步骤。

## 测试要求

- 后端：
  - 新增或扩展 `pytest` 用例，验证静态资源与 SPA fallback 行为。
  - 回归执行 `pytest recall/tests/test_api.py recall/tests/test_regression_pipeline.py`。
- 前端：
  - 执行 `npm run build`，确认构建产物可被 FastAPI 正常托管。
- 联调：
  - 以“仅启动 FastAPI”为条件，人工验证页面可访问、关键 API 可用。
