## 背景

重构目标要求先聚焦本地 Recall 主链路，触发截图、落盘与资源监控是 OCR 前的前置能力。

## 目标

实现从“触发条件达成”到“截图落盘 + 元数据入库”的可运行链路，并提供资源空闲状态输出给 OCR 调度使用。

## 范围

包含：
- `trigger`：变化阈值与强制间隔触发策略。
- `capture`：执行截图、按日期路径落盘、写入 `screenshots`（`ocr_status=pending`）。
- `resource-monitor`：采集 CPU/GPU 使用率并输出繁忙/空闲状态。

不包含：
- OCR 推理与文本回写。

## 实施计划（分步骤）

1. 定义 trigger/capture/monitor 模块接口与事件总线。
2. 实现截图触发判定和节流策略。
3. 实现截图文件落盘规则：`screenshots/YYYY-MM-DD/HH/{id}.jpg`。
4. 写入截图元数据（时间、路径、窗口信息、phash、状态）。
5. 实现资源监控周期任务并暴露状态查询接口给内部模块。

## 验收标准

- [x] 触发策略可稳定产出截图事件。
- [x] 截图文件可按约定目录落盘且数据库记录完整。
- [x] 资源监控能输出可用于 OCR 调度的繁忙/空闲状态。
