## 背景

OCR 技术路线已固定为 `onnxruntime-node` 自建能力，需与 `screenshots` 待处理队列和资源监控联动。

## 目标

实现可持续运行的 OCR Worker：批量读取 `pending` 截图，执行 OCR，回写文本与状态。

## 范围

包含：
- OCR Worker 任务循环与批处理参数（阈值、批大小）。
- `onnxruntime-node` 推理调用封装。
- OCR 结果回写：`ocr_text`、`ocr_status(done/error)`、错误信息记录。
- 与 `resource-monitor` 状态联动（仅在资源允许时执行）。

不包含：
- 前端 OCR 展示逻辑。

## 实施计划（分步骤）

1. 定义 OCR 队列查询与状态流转规则。
2. 集成 `onnxruntime-node`，封装模型加载与推理接口。
3. 实现批处理循环（拉取 pending -> 推理 -> 回写结果）。
4. 接入资源状态判断，繁忙时退避重试。
5. 增加错误处理、指标日志和基础测试。

## 验收标准

- [x] pending 截图可被批量处理并回写 OCR 文本。
- [x] 出错截图可标记 `error` 且不阻塞后续任务。
- [x] OCR 执行会受资源监控状态控制。

## 测试要求

- [x] OCR 队列状态流转单元测试。
- [x] Worker 批处理与错误分支集成测试（可使用 mock 模型）。
- [ ] 手工验证连续运行稳定性（至少一个完整周期）。
