## 已修复

### BUG: 对话中调用工具会卡住 (v0.6.3)

**现象**：对话中调用工具会卡住，只回复文字没问题。前端显示永久的"回复中..."状态。

**原因**：`curious_ai.py` 的 `_process_response()` 方法在执行工具后没有继续调用 LLM：
- 缺少工具调用循环：执行完工具后直接结束，没有将工具结果反馈给 LLM
- 消息格式不完整：没有将 assistant 的 tool_calls 和 tool 结果添加到对话历史
- 前端卡住：pending 消息永远不会被更新为 done 状态

**修复**：参考 `llm.py` 的正确实现，在 `curious_ai.py` 中添加完整的工具调用循环：
- 重构 `_handle_user_message()` 方法，添加 while 循环处理工具调用
- 将 assistant 消息（包含 tool_calls）添加到对话历史
- 将工具执行结果作为 `role: tool` 消息添加到对话历史
- 再次调用 LLM 直到没有工具调用或达到最大迭代次数（3次）
- 同样修复 `_explore()` 方法的工具调用处理
- 删除不再需要的 `_process_response()` 方法

**测试**：新增 `tests/unit/test_curious_ai.py`，覆盖 8 个测试用例

### BUG: AI不回复用户消息 (v0.6.2)

**现象**：保存配置后，再次进入设置页面显示的是旧值

**原因**：配置存储在 Python 文件中，使用正则表达式替换不可靠，且 `importlib.reload` 在多进程环境下不同步

**修复**：重构配置系统
- 配置改为存储在 SQLite 数据库的 `settings` 表中
- 使用 `config.get(key)` 从数据库读取配置
- 主应用和 Web 服务共享数据库，配置实时同步
- 支持热更新，修改配置后立即生效

### BUG: 聊天界面不显示用户输入的消息 (v0.6.1)

**现象**：用户在聊天界面发送消息后，消息不会立即显示，需要等待轮询（约1秒）才能看到

**原因**：`handleSend()` 函数在成功发送消息后没有将用户消息添加到 `messages` 数组，完全依赖轮询机制从后端获取

**修复**：采用乐观更新策略
- 发送时立即显示用户消息（使用临时 ID）
- 发送成功后更新为真实 ID
- 发送失败时移除临时消息并显示错误提示
- 改进 `pollMessages()` 去重逻辑，同时检查 ID 和内容，避免消息重复

### BUG: 托盘图标非常驻，关闭前端窗口后程序退出 (v0.5.5)

**现象**：使用 scripts 启动程序后，弹出前端窗口，关闭窗口后整个后台程序也一起退出

**期望**：托盘图标应常驻，关闭前端窗口后程序继续在后台运行，通过托盘图标可以重新打开窗口或退出程序

**修复**：
- 添加 `_on_window_closing()` 方法，拦截窗口关闭事件，改为隐藏窗口
- 托盘线程改为非守护线程，确保程序不会意外退出
- 只有通过托盘菜单点击"退出"才会真正退出程序
